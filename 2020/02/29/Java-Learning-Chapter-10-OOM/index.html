Java Learning Chapter 10 Object-Oriented Modelling

Author: Domuki Date: February 29, 2020  18:32:49 Category: Java Study

Learning Objectives

Explain the process of object-oriented software development (A3)

Analyze the type of relationship: association, aggregation, strong inheritance, and weak inheritance (C4)

1. Software Development Process

1. Requirement Specification

A formal process that seeks to understand the problem and document in detail what the software system needs to do. This phase involves close interaction between users and designers.

Most of the examples in this book are simple, and their requirements are clearly stated. In the real world, however, problems are not well defined. You need to study a problem carefully to identify its requirements.

2. System Analysis

Seeks to analyse the business process in terms of data flow, and to identify the system’s input and output.

Part of the analysis entails modelling the system’s behaviour. The model is intended to capture the essential elements of the system and to define services to the system.

3. System Design (What’s the dream looks like)

The process of designing the system’s components.

This phase involves the use of many levels of abstraction to decompose the problem into manageable components, identify classes and interfaces, and establish relationships among the classes and interfaces.

4. Implementation (From dream to reality)

The process of translating the system design into programs. Separate programs are written for each component and put to work together.

5. Testing (Find problems)

Ensures that the code meets the requirements specification and weeds out bugs.

An independent team of software engineers not involved in the design and implementation of the project usually conducts such testing.

6. Deployment (Make program can be use)

Deployment makes the project available for use.

For a Java program, this means installing it on a desktop or on the Web.

7. Maintenance

Maintenance is concerned with changing and improving the product.

A software product must continue to perform and improve in a changing environment. This requires periodic upgrades of the product to fix newly discovered bugs and incorporate changes.

3. Class Relationships

1. Association

Association represents a general binary relationship that describes an activity between two classes. An association is usually represented as a data field and methods in the class.

2. Composition

We refer aggregation between two objects as composition if the existence of the aggregated object is dependent on the aggregated object.

If the whole is destroyed, the parts must also be destroyed together, and if the owner is copied, the parts must also be copied together.

For example, “a body has a heart” is a composition, the heart class depends on body class .The aggregated object can’t exist on it’s own.

Another example is “a student has a name”

3. Aggregation (has-a)

Aggregation is a special form of association that represents an ownership relationship between two object.

Both students and schools can exist on their own without each other.

4. Inheritance

Keyword: extends

Inheritance (Generalization) is the relationship between the parent class and the child class. The child class inherits all the structure and behaviour of the parent class. New structures and behaviours can be added to subclasses, and the behaviour of parent classes can also be overridden.

5. Dependency

Dependency is a usage relationship. The change of a particular thing may affect the thing that uses it. Use when you want to show that one thing uses another.

6. Realization

Keyword: implements

The realization relationship specifies a contract between two entities. In other words, one entity defines a contract, and another entity guarantees the performance of the contract. For a class, a class implements an interface.

7. Summary

Four types according to UML recommendations:

  1. Dependency

  2. Association

  3. Aggregation

  4. Composition

Strong and weak relationship:

​ dependent < association < aggregation < composition < generalization (inheritance)

Differences in implementation:

(1) Dependency:

​ The relationship object appears in the parameters of local variables or methods, or the static method of the relationship class is called

(2) Association relationship:

​ relationship objects appear in instance variables

(3) Aggregation relationship:

​ relationship objects appear in instance variables

(4) Composite relationship:

​ relationship objects appear in instance variables

(5) Generalization: extends

(6) Implementation: implements
